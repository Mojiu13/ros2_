# MODULE_INTERVIEW_RULES｜模块面试提问规则

本文件定义所有学习模块结束前必须执行的“模块面试提问环节”。

这个环节不是最后求职前才做，而是每个模块的验收方式之一。

---

## 1. 总原则

每个模块结束前，必须加入一次模块面试。

目的不是背答案，而是检查：

```text
我是否真的理解了本模块？
我能否不用看代码讲清楚？
我能否把概念、命令、现象、故障和项目表达串起来？
```

不要等到全部学完后才开始面试复盘。

正确节奏是：

```text
每个模块末尾：模块小面试
每个阶段结束：阶段综合面试
全部学完后：JOB-M3 总面试库 + 模拟面试
```

---

## 2. 每个模块末尾必须执行的流程

模块结束前，请执行以下流程：

```text
1. 根据当前模块内容，先问我 5～8 个面试问题；
2. 不要先给标准答案；
3. 让我先逐题回答；
4. 根据我的回答指出：
   - 概念漏洞
   - 表达问题
   - 工程理解不足
   - 排查顺序错误
5. 让我再复述一次关键答案；
6. 最后整理成可写入 docs/INTERVIEW_QA.md 的标准问答；
7. 如果本模块产生了项目代码，也要加入项目表达题。
```

---

## 3. 模块面试必须覆盖的 5 类问题

每次模块面试都要尽量覆盖以下 5 类问题。

### 3.1 概念边界题

问“它是什么，不是什么”。

示例：

```text
topic 是什么？
service 是什么？
action 是什么？
URDF 是什么？TF 是什么？
MoveIt 负责什么？不负责什么？
controller 负责什么？不负责什么？
```

---

### 3.2 系统角色题

问“它在完整机器人系统中扮演什么角色”。

示例：

```text
robot_state_publisher 做什么？
joint_state_publisher 做什么？
move_group 做什么？
controller_manager 做什么？
joint_trajectory_controller 在执行链路中处于什么位置？
```

---

### 3.3 CLI 观察命令题

问“用什么命令观察”。

示例：

```text
如何查看当前有哪些 node？
如何查看 topic 消息？
如何查看 action 是否存在？
如何查看 controller 是否 active？
如何查看 /move_group 参数？
如何查看 TF tree？
```

---

### 3.4 故障排查题

问“坏了怎么办”。

示例：

```text
节点没出现，先查什么？
topic 没消息，先查什么？
RViz 不显示机器人，先查什么？
Plan 成功但 Execute 失败，先查什么？
controller inactive，怎么定位？
joint name mismatch 可能发生在哪些层？
```

---

### 3.5 项目表达题

问“你怎么给面试官讲”。

示例：

```text
这个模块你做了什么？
你怎么证明它跑通了？
你遇到了什么错误？
你怎么定位？
这个模块和完整机械臂执行链路有什么关系？
如果写进简历，这个模块体现了什么能力？
```

---

## 4. 不同阶段的提问深度

### R0 / R1 阶段

以基础理解和链路感为主。

重点问：

```text
这个东西叫什么？
它在哪一层？
用什么命令观察？
它和完整链路有什么关系？
失败时先看哪里？
```

不要过早追问源码、OMPL、IK、实时控制等深层细节。

---

### R2 阶段

以工程配置和故障排查为主。

重点问：

```text
配置文件怎么对应？
参数从哪里加载？
joint name 如何贯穿 URDF、SRDF、controller.yaml？
action namespace 配错会怎样？
controller failed to activate 怎么查？
time_from_start 为什么重要？
```

---

### MP 阶段

以项目答辩为主。

重点问：

```text
你的项目解决了什么问题？
你为什么这样封装？
相比官方 demo，你多做了什么？
如果任务失败，系统如何处理？
这个项目能证明你哪些工程能力？
```

---

### JOB 阶段

以真实面试和表达包装为主。

重点问：

```text
请用 3 分钟介绍你的项目。
请画出 MoveIt 到 controller 的执行链路。
如果机械臂不动，你按什么顺序排查？
你对 joint_trajectory_controller 的理解到什么程度？
你项目中最有含金量的故障排查是什么？
```

---

## 5. 输出要求

模块面试结束后，必须输出两份内容。

### 5.1 给学习者的反馈

格式：

```text
你答得好的地方：

你暴露的问题：

需要重新复述的关键点：

下一模块开始前必须补齐的理解：
```

### 5.2 写入 docs/INTERVIEW_QA.md 的标准问答

格式：

```markdown
## <MODULE_ID>｜<MODULE_NAME>

### Q1：...

回答：
...

### Q2：...

回答：
...
```

---

## 6. 模块完成判定

一个模块不能只因为“代码跑通”就算完成。

必须同时满足：

```text
1. 最小例子跑通；
2. 会用 CLI 观察；
3. 至少记录一个关键现象或故障；
4. 能画出或说出本模块在系统链路中的位置；
5. 完成模块面试；
6. 生成可写入 docs/INTERVIEW_QA.md 的问答。
```

如果第 5、6 步没做，本模块不算真正完成。
